Issue Register — PICA Workflow
Problem Identification & Corrective Action · Project issue tracking, escalation management, and close-out monitoring for the CHPP2 EPC Project at Lampunut Mine, Central Kalimantan.
Document: CHPP2-MOD14-PICA-001 Rev A Date: March 2026 Module Type: Issue Management Data Source: ISSUE_PICA_Register (Google Sheets) Update Freq.: On Event / Weekly Review
Issue Register vs Risk Register — Critical Distinction
The Risk Register (Module 06) tracks potential future events that may or may not occur, assessed by probability and impact. The Issue Register (Module 14) tracks problems that are already happening now — confirmed, active problems requiring immediate response and corrective action. Both registers are maintained in parallel. When a risk materialises, it is formally transferred to the Issue Register and assigned a PICA record.
1
RISK REGISTER vs ISSUE REGISTER
Conceptual separation — how both registers co-exist in the CHPP2 dashboard
Dimension Risk Register — Module 06 Issue Register — Module 14 (PICA)
Definition Potential event that may occur and affect project objectives. Has probability > 0 and < 1. Problem that has already occurred or is actively occurring. Probability = 1 (it is real).
Time orientation Future-looking. Preventative and contingency planning. Present / ongoing. Reactive and corrective response required now.
Key metrics Probability × Impact score. Risk heat map. Residual risk rating. Days open. Overdue actions. Root cause category. Corrective action effectiveness.
Owner Risk Owner — responsible for mitigation plan. Action Owner — responsible for corrective action and close-out.
Response type Avoid / Transfer / Mitigate / Accept. PICA: Identify problem → Analyse impact → Define corrective action → Verify effectiveness.
Resolution Risk closed when probability drops to 0 or event window passed. Issue closed only after corrective action verified effective. Requires PMC sign-off.
Transfer rule When a risk event materialises, it moves from Risk Register → Issue Register. Risk entry marked "Materialised". Receives cross-reference to originating Risk ID. Inherits risk category and impact class.
CHPP2 examples Risk of DADI delivery delay due to wet season (Dec–Feb). Risk of Chinese steel certification rejection at customs. Civil engineering drawings 45 days behind schedule (confirmed — happening now). Electrical design criteria change request unresolved for 30 days.
Materialisation trigger: When a risk event is confirmed to have occurred, the PMC formally transfers it to the Issue Register within 48 hours. The Risk Register entry is updated with status "Materialised — See PICA-XXXX" and probability is set to 1.0.

PICA FRAMEWORK — Four-Phase Structure
Problem Identification & Corrective Action — each issue passes through all four phases in sequence
P — Problem
Problem Identification
What happened? Where? When?
Issue ID Required
Auto-generated sequential number. Format: PICA-CHPP2-YYYY-NNN
Problem Statement Required
Clear, factual description of the problem. Answers: what is wrong, where, since when. No blame attribution.
Category Required
Engineering / Procurement / Logistics / Construction / Commercial / HSE / Quality / Contractual / Regulatory
Location / WBS Reference Required
Specific WBS ID and physical location or contract scope area affected.
Date Identified / Raised By Required
Date issue was first identified. Party and individual who raised it.
Priority Required
Critical / High / Medium / Low — based on impact assessment in Phase I.
Originating Risk ID Optional
Cross-reference to Risk Register entry if this issue materialised from a tracked risk.
I — Impact
Impact Assessment
How bad? What is affected?
Schedule Impact Required
Estimated delay in calendar days. State whether on critical path. Cross-reference to Gantt (Module 13).
Cost Impact Required
Estimated additional cost in USD. If recoverable through VO, state VO reference. If unrecoverable, state exposure.
Scope Impact Required
Which deliverables, systems, or contract milestones are affected. None / Minor / Moderate / Major.
Safety / Regulatory Impact If applicable
AMDAL compliance, TKDN compliance, ESDM permit status, or HSE incident trigger.
Affected Parties Required
Which contract parties are directly affected: EP Contractor / Construction Contractor / PMC / Employer.
Interface Impact If applicable
Impact on EP–CC interface hand-off points. Equipment delivery to site, shop drawing review loops.
Root Cause Category Required
Design / Procurement / Logistics / Resource / Communication / External / Contractual / Process / Unknown
C — Corrective
Corrective Action
What will be done? By whom? By when?
Immediate Containment Action Required
Short-term action taken within 24–72 hours to stop the problem from worsening. Does not fix root cause.
Corrective Action Description Required
Full description of the action that will permanently resolve the issue and address the root cause.
Action Owner Required
Named individual (not just party/company) responsible for delivering the corrective action.
Target Completion Date Required
Agreed deadline for corrective action completion. Overdue flag triggers if exceeded.
Resources Required If applicable
Additional manpower, equipment, cost, or external expertise needed to execute the corrective action.
VO / EOT Reference Optional
If the corrective action requires a Variation Order or Extension of Time, cross-reference to VO_ChangeLog (Module 07).
Progress Notes Required
Weekly progress updates on corrective action execution. Date-stamped log appended each update cycle.
A — Accountability
Verification & Close-Out
Did it work? Is it closed?
Verification Method Required
How will effectiveness of the corrective action be confirmed: inspection, test, document review, KPI measurement, or third-party witness.
Verification Owner Required
PMC / Employer's Engineer is the mandatory verifying authority for all Priority Critical and High issues. Medium/Low may be verified by PMC Quality.
Actual Completion Date When complete
Date corrective action was physically completed and verified. Must be recorded before close-out.
Effectiveness Confirmed Required
Yes / No / Partially — PMC sign-off confirming the issue has been fully resolved and will not recur.
Preventive Action For Critical / High
Systemic change to prevent recurrence: process update, specification revision, training, or procedure change.
Lessons Learned Optional
Summary for project knowledge base. Flagged for inclusion in monthly project review and shared with Adaro Group.
Close-Out Authorisation Required
PMC / Employer's Engineer signature (name + date) confirming formal close-out. Dashboard status changes to "Closed" only on this event.
Sequential progression rule: A PICA record must complete Phase P and Phase I before Corrective Action (Phase C) is initiated. Close-Out (Phase A) can only be triggered after corrective action is implemented and PMC has physically verified effectiveness. No phase can be skipped, regardless of priority level.

Priority-based response SLAs, ownership, and escalation chain for the CHPP2 project hierarchy

Critical

Containment action		 EP / CC Project Director + PMC Project Manager >3 days → Employer Project Director >7 days → Adaro Group Board Daily status update until resolved

Low
Minor issue with minimal project impact. Administrative or cosmetic in nature. Document formatting non-conformance. Minor site housekeeping. Label change request. 10 working days
Action initiated		 Relevant discipline officer >21 days → PMC Quality >30 days → PMC Lead Engineer Monthly register review
Critical escalation — Employer obligation
Under the CHPP2 Engineering & Procurement Contract, the Employer (PT Maruwai Coal) must be notified of any Critical issue within 24 hours of identification, regardless of which party raises it. The PMC / Employer's Engineer is responsible for ensuring this notification occurs. Failure to notify constitutes a reporting non-conformance under Contract Clause 5.3 (Employer's Representative obligations).

GOOGLE SHEETS DATA SCHEMA — ISSUE_PICA_Register
Complete column specification for the master data source. Column names are locked — never rename.
# Column Name PICA Phase Data Type Required Description & Validation Rules
1 PICA_ID System Text (auto) Auto Auto-generated on row creation. Format: PICA-CHPP2-{YYYY}-{NNN} e.g. PICA-CHPP2-2026-001. Immutable once set. Primary key.
2 Date_Raised P Date (DD/MM/YYYY) Required Date the issue was first identified and formally raised. Cannot be a future date.
3 Raised_By_Name P Text Required Full name of individual who identified and raised the issue.
4 Raised_By_Party P Dropdown Required Dropdown: EP Contractor / Construction Contractor / PMC / Employer. Party that identified and submitted the issue.
5 Problem_Statement P Text (long) Required Clear factual description of the problem. Minimum 30 characters. Must answer: what, where, since when. Validated: no blame language.
6 Category P Dropdown Required Engineering / Procurement / Logistics / Construction / Commercial / HSE / Quality / Contractual / Regulatory / Interface
7 WBS_Reference P Text Required WBS ID from GANTT_Schedule_Master (e.g. 1.2.2). Cross-links to Gantt (Module 13) and S-Curve (Module 01).
8 Priority P Dropdown Required Critical / High / Medium / Low. Drives escalation rules and SLA timers. PMC may upgrade/downgrade priority during review.
9 Originating_Risk_ID P Text Optional Cross-reference to RISK_IssueRegister (Module 06) if this issue materialised from a tracked risk. Format: RISK-CHPP2-NNN.
10 Schedule_Impact_Days I Integer Required Estimated delay in calendar days. Enter 0 if no schedule impact. Negative values not permitted.
11 On_Critical_Path I Dropdown Required Yes / No / Unknown. If Yes, auto-upgrades minimum priority to High and flags on dashboard critical path view.
12 Cost_Impact_USD I Number Required Estimated additional cost in USD. Enter 0 if no cost impact. Links to Commercial module (Module 08) VO tracking if applicable.
13 VO_Reference I Text Optional Cross-reference to VO_ChangeLog (Module 07) if issue is linked to an active variation order. Format: VO-CHPP2-NNN.
14 Root_Cause_Category I Dropdown Required Design / Procurement / Logistics / Resource / Communication / External / Contractual / Process / Unknown. Used for trend analytics.
15 Root_Cause_Narrative I Text (long) Cond. Required for Priority Critical and High. Narrative explanation of why the problem occurred. Supports lessons learned.
16 Affected_Parties I Multi-select text Required Comma-separated list: EP Contractor, Construction Contractor, PMC, Employer. All directly impacted parties.
17 Containment_Action C Text (long) Required Short-term containment action taken immediately to prevent escalation. Required within 48h of issue opening. Distinct from corrective action.
18 Corrective_Action_Description C Text (long) Required Full description of the permanent corrective action. Must address root cause. Minimum 50 characters for Priority Critical/High.
19 Action_Owner_Name C Text Required Named individual responsible for delivering corrective action. Must be a specific person, not a company or team name.
20 Action_Owner_Party C Dropdown Required EP Contractor / Construction Contractor / PMC / Employer. Party responsible for executing the corrective action.
21 Target_Date C Date (DD/MM/YYYY) Required Agreed target completion date for corrective action. Must align with Priority SLA. Dashboard overdue flag fires when date is exceeded.
22 Progress_Notes C Text (append-only log) Required Weekly appended progress log. Format: [DD/MM/YYYY — Name]: Progress note text. Do not overwrite prior entries. Concatenated log in one cell.
23 Pct_Complete_Action C Integer (0–100) Required Estimated % completion of corrective action execution. Updated weekly by Action Owner. Used in dashboard progress bar visualisation.
24 Verification_Method A Text Required How effectiveness will be confirmed: Physical Inspection / Document Review / Test / KPI Measurement / Third-Party Witness.
25 Verification_Owner_Name A Text Required Named PMC individual responsible for verifying corrective action effectiveness. Must be PMC / Employer's Engineer role for Critical and High priorities.
26 Actual_Completion_Date A Date (DD/MM/YYYY) Cond. Populated only when corrective action is physically complete. Must be present before status can move to "Pending Verification".
27 Effectiveness_Confirmed A Dropdown Cond. Yes / Partially / No. Populated by PMC Verification Owner. "Partially" or "No" returns issue to "Action In Progress".
28 Preventive_Action A Text (long) Cond. Required for Critical and High priorities. Systemic change to prevent recurrence. Logged in Lessons Learned library.
29 Lessons_Learned A Text Optional Key insight for future reference. Exported monthly to Adaro Group knowledge management platform. Visible on Dashboard read-only.
30 Closeout_Auth_Name A Text Cond. Name of authorising signatory for formal close-out. PMC Project Manager for Critical/High. PMC Lead Engineer for Medium/Low.
31 Closeout_Date A Date (DD/MM/YYYY) Cond. Date of formal close-out authorisation. Dashboard status changes to "Closed" only when this field is populated. Immutable once entered.
32 Current_Status System Dropdown (managed) Auto Open / Under Review / Action In Progress / Escalated / Pending Verification / Resolved / Closed. See allowed transitions table.
33 Days_Open System Integer (formula) Auto Formula: =TODAY() - Date_Raised (or Closeout_Date if closed). Drives overdue flags and SLA breaches.
34 Days_Overdue System Integer (formula) Auto Formula: =MAX(0, TODAY() - Target_Date). Positive value triggers overdue badge and escalation alert.
35 Last_Updated System Datetime (auto) Auto Auto-updated timestamp each time row is modified. Used to identify stale records with no progress update for >7 days.
Sheet protection: Columns 1 (PICA_ID), 32 (Current_Status — final close only), 33–35 (formula columns) are sheet-protected. Users may not modify PICA_ID or formula outputs directly. Status transitions are managed through the dashboard UI which validates allowed transitions before writing back to the sheet.

INTEGRATION WITH OTHER DASHBOARD MODULES
Module 14 is a cross-cutting module — issues link to and from six other modules in the CHPP2 dashboard
Module 06
Risk Register
Risk → Issue
When a risk materialises, it transitions to a PICA record. Originating_Risk_ID field cross-links. Risk entry marked "Materialised" in Module 06.
Module 13
Gantt Chart
Issue → Schedule delay
Schedule_Impact_Days and WBS_Reference fields drive automatic delay flags on the Gantt chart. Issues on critical path are highlighted in Module 13.
Module 07
Variation Orders
Issue → VO raised
VO_Reference field links PICA records requiring contract variation to the VO_ChangeLog. Cost_Impact_USD feeds into commercial exposure tracking in Module 08.
Module 01
S-Curve Progress
Issue delay → SPI impact
Schedule impacts from open Critical/High issues are reflected in the earned value computation. SPI deterioration is traceable to specific PICA records.
Module 09
HSE Dashboard
HSE event → Issue raised
Any HSE incident above near-miss level automatically generates a PICA record with Category = HSE. Corrective action close-out is mandatory before LTI-free counter resets.
Homepage
Executive Summary
Issue KPIs → Board view
Open Critical issues, escalation count, and overdue action count appear on the Homepage RAG scorecard. Any Escalated issue triggers the Homepage alert banner.
Governance note: The ISSUE_PICA_Register Google Sheet is owned by the PMC / Employer's Engineer. EP Contractor and Construction Contractor may raise new issues and update their own corrective action progress fields only. PMC holds exclusive write access to status transitions, verification fields, and close-out authorisation. Employer holds read-only access to all fields at all times via the dashboard.
