Core Architecture Principle

Excel is the master. Firebase is the mirror. The dashboard is the screen. Data only flows one direction: Excel → Power Automate → Firebase → Dashboard. The dashboard never writes back to Excel. The only exception is Risk Register and Issue Register, which write directly to Firebase from the dashboard interface.

📥 Data Entry Points
  • Excel on OneDrive/Teams — Tasks, S-Curve, Procurement, Budget
  • Dashboard inline forms — Risk Register, Issue Register only
  • Firebase Console — Admin/emergency corrections only
📤 Data Flow
  • Excel save → Power Automate trigger (new rows, instant)
  • Scheduled full-sync daily at 6AM (catches edits)
  • Risk/Issue forms → Firebase directly (real-time)
  • Dashboard reads Firebase via onSnapshot() (live)

Real-time database connection — data updates the dashboard within seconds
📁 Firestore Collection Structure

⚡ Firebase SDK Functions Required
  • getTasks(), getRisks(), getIssues()
  • getProcurement(), getBudget(), getSCurve()
  • addDocument(collection, data)
  • updateDocument(collection, id, data) — with merge: true
  • deleteDocument(collection, id)
  • onSnapshot() — real-time live listeners
🔧 Claude Code Prompt for Firebase Connection

Ask Claude Code: "Add Firebase Firestore SDK v10 (modular) from CDN. Connect using [paste your config]. Create a firebaseService object with all CRUD functions. Use onSnapshot() for real-time listeners on all collections. Show 'Last updated: X seconds ago' in the header."

⚠️
Critical: Use merge:true on all updates

Always use setDoc(docRef, data, { merge: true }) — never plain setDoc(docRef, data). Without merge:true, saving partial data overwrites and erases fields you didn't include in the update.

Excel File Rules (Critical for Power Automate Compatibility)

✅
Always Do
  • Convert all data ranges to named Tables (Ctrl+T)
  • Use underscores in column headers: Task_Name not "Task Name"
  • Add new columns at the far right end only
  • Use DD/MM/YYYY date format consistently
  • Fill all cells in existing rows when adding a new column
❌
Never Do
  • Insert columns in the middle of a table
  • Delete rows — mark as Cancelled/Archived instead
  • Use spaces or special characters in column headers
  • Leave blank rows between data rows
  • Use % symbol in number fields (just use the number)

⚡ Power Automate Flow Structure (Each Flow)
  1. Trigger: "When a row is added (Excel Online Business)" → select file and table
  2. Compose action: Build JSON body mapping Excel columns to Firebase fields (use integerValue for numbers, stringValue for text)
  3. HTTP action: PATCH to Firebase REST API — https://firestore.googleapis.com/v1/projects/[project-id]/databases/(default)/documents/[collection]/[row-id]?key=[api-key]
  4. Error handling: Condition → if HTTP status ≥ 400 → send Outlook email alert
💡
Null-safe field expression in Power Automate

Wrap all fields to prevent crashes on empty cells: @{if(empty(triggerOutputs()?['body/Column_Name']), '', triggerOutputs()?['body/Column_Name'])}

10. Schema Change Management

How to safely add new columns to existing modules without breaking anything
📌
The Golden Rule Before Any Schema Change

Message your team: "Pausing [module] pipeline for 30 minutes." Turn off the affected Power Automate flow. Backup index.html. Then proceed.

📋 The 4-Layer Change Impact — Every new column touches all 4
Excel
Add column at far right
→
Power Automate
Add JSON field mapping
→
Firebase
Backfill via full sync
→
Dashboard
Add to table + edit form

Step-by-Step for Every Column Addition

1
Excel — Add the New Column
  • Click the header of the LAST column → press Tab to add new column to the right
  • Name with underscores: Review_Date not "Review Date"
  • Fill in a value for EVERY existing row (even if "N/A") — blank breaks Power Automate
  • Save the file
2
Power Automate — Add JSON Field + Null Protection

Open the Compose action → add ONE new line only:

"review_date": {"stringValue": "@{if(empty(triggerOutputs()?['body/Review_Date']), '', triggerOutputs()?['body/Review_Date'])}"}

For number fields use integerValue and wrap with int():

"score": {"integerValue": "@{int(triggerOutputs()?['body/Score'])}"}

Save → Turn flow back ON → Test → verify green ✓

3
Firebase — Backfill Existing Documents

Run the Full Sync scheduled flow manually RIGHT NOW. This re-syncs every row including the new column. All existing Firebase documents will have the new field within minutes. Never do this manually document by document.

4
Dashboard — Add to Table Display + Edit Form

Ask Claude Code: "In my [module] section, I added a new Firebase field called 'review_date'. Please: (1) add a 'Review Date' column to the table, (2) add a 'Review Date' input to the Add/Edit side panel form, (3) ensure updateDocument() includes the new field. Here is the current code: [paste module code]"

5
Check Executive Summary + Deploy + Document

Check if the new column affects any KPI cards in Executive Summary. Run the 5-minute regression test. Deploy via firebase deploy. Update README.txt change log. Notify team.

11. Access Control & Security

Simplified — the dashboard is read-only, so requirements are minimal
✅
Key Decision: Role-Based Access is NOT needed

Because all data entry happens in Excel and the dashboard is 100% read-only, there is nothing to protect from being edited through the web interface. Role-based access control (Admin/Editor/Viewer) is only needed when users can create or modify data through the web app — which this dashboard does not allow (except Risk and Issue registers, managed by your own team).

🔐 Security Decision Matrix — Choose One Row
ScenarioWhat You NeedEffort
Internal team, low-sensitivity dataSimple password on login screen (already built)✅ Done
Client needs access but not cost/risk detailsTwo HTML files: full version + client-facing version (remove sensitive modules)⭐ Low
Highly confidential commercial dataFirebase Google Sign-In restricted to @yourcompany.com email⚠️ Medium
Public-facing progress dashboardNo login at all — share URL directly✅ Done
📌
OneDrive/SharePoint already handles data security

Access to edit Excel files is controlled by Microsoft Teams channel membership. Team members = can edit. Guests/clients added to Teams = set read-only on the Files tab. People outside Teams = no access to Excel at all. Nothing additional needed on the dashboard side for data integrity.

12. Data Entry Method — Final Decision

Hybrid approach matching entry method to data type and user behavior
🎯
The Hybrid Principle

Use Excel for structured bulk data that's managed in batches. Use the dashboard web interface for event-driven records that are logged in real-time and updated continuously through a lifecycle.

ModuleMethodReason
Tasks / Gantt ✅ Excel Inherently tabular. Planning engineers manage this in Excel naturally. Batch updates weekly.
S-Curve Progress ✅ Excel Small monthly table (12–24 rows/year). Numbers calculated externally before entry. Excel is the right tool.
Procurement ✅ Excel Wide rows with many columns updated together. Procurement officer's natural workflow is spreadsheet-based.
Budget / Cost ✅ Excel Data comes from accounting/ERP systems. Cost controller processes figures in Excel first regardless. No benefit to a web form.
Risk Register ⭐ Dashboard Interface Risks are logged one at a time in real-time (meetings, site walks). Frequently updated through lifecycle (likelihood changes, mitigation updates, status to Closed). Web interface is 10x faster for these micro-updates.
Issue Register ⭐ Dashboard Interface Same as risks — issues happen in the moment. Must be logged immediately. Status changes frequently: Open → In Progress → Resolved → Closed. Web form is the right tool.
⚠️
Why not Microsoft/Google Forms?

Forms can only add new records — they cannot edit existing ones. For Risk and Issue registers where records go through a multi-stage lifecycle with frequent updates, forms create duplicate/confusing entries. The dashboard inline edit panel handles both adding AND updating in one clean interface.

🔧 What This Means for Your Dashboard Build

For Risk Register and Issue Register only, the dashboard needs direct Firebase write capability:

  • Floating + button (bottom-right) → opens empty add form → saves to Firebase
  • Clicking any table row → opens pre-filled edit panel → saves changes with merge: true
  • Delete button with confirmation dialog
  • Cancel button closes panel without saving

All other modules remain 100% read-only on the dashboard.

13. Additional Considerations

Important factors to plan for as the project evolves
⚡ Performance (as data grows)
  • Add pagination to all tables — 25 rows per page with Previous/Next buttons
  • Use Firebase query filters (where clause) instead of fetching all data and filtering in JavaScript
  • Gantt Chart: if 200+ tasks, add "Load by Phase" toggle
  • S-Curve: maximum 24 data points (monthly) is fine
📡 Offline Resilience
  • Enable Firebase offline persistence — dashboard loads cached data when no internet
  • Show "⚠️ Offline — showing last synced data" banner when connection drops
  • Power Automate retry policy: 5 retries, 10-minute intervals — configure on each flow
  • Critical for teams on construction sites with poor connectivity
🖨️ Reporting & Export
  • Export to Excel button on each table (SheetJS library, free CDN)
  • Print-friendly CSS stylesheet — hides sidebar and buttons, prints clean A4
  • Scheduled Power Automate email: every Monday 8AM → reads Firebase → sends HTML summary email via Outlook
📋 Audit Trail
  • Add last_modified_by and last_modified_at fields to every Firebase document
  • Power Automate: include x-ms-user-displayname header in every sync
  • Optional: change_log Firestore collection recording what changed, who, when, old vs new value
💰 Cost Management
ServiceFree LimitWatch For

Firebase Hosting10GB/monthVery unlikely to hit

📅 Project Lifecycle
  • Plan for phase transitions: Pre-Project → Active → Completion → DLP → Closed
  • Never delete records — mark Status as "Archived" or "Closed"
  • Freeze S-Curve at 100% on completion (don't allow backwards movement)
  • Multi-project: prefix all IDs with project code (e.g., ALPHA-TSK-001) for future scalability
🧪 5-Minute Regression Test (Run After Every Change)
  1. Open dashboard → navigate to the changed module
  2. Does the table load with all columns (including any new ones)?
  3. Do all filters and search still work?
  4. Add a new record → does all data save correctly to Firebase?
  5. Edit an existing record → does it update without wiping other fields?
  6. Check Executive Summary — any broken KPI cards?
  7. Open on mobile — does the new column display correctly?

If all 7 pass → deploy. If any fail → fix first.
